# research-atlas — 어떻게 동작하나 (HOW IT WORKS)

> 각 기능이 *무엇을* 하고 *어떻게* 동작하는지. 쉽고 자세하게.
> README가 "1페이지 요약"이라면, 이 문서는 그 밑의 "메커니즘 전체"다.

---

## 0. 큰 그림

research-atlas는 두 부분이다.

```
[빌드 파이프라인]  논문 → 개념·계보 추출 → 그래프 DB(Neo4j)에 적재
                   (오프라인, 가끔 돌림)

[웹 앱]            그 그래프를 읽어 화면에 그리고, 질문하면 반응
                   (FastAPI 백엔드 + React 프론트, 항상 떠 있음)
```

용어 셋만 먼저:
- **개념(concept) 노드** = 하나의 기법/방법 (예: RAG, ReAct). 그래프의 점.
- **builds_on 엣지(계보)** = "A 방법이 B를 딛고 섰다"는 방법적 후예 관계. 그래프의 선. 점수 비교용 baseline·부품은 **제외**(lineage-only).
- **사전(lexicon)** = 개념의 "상태 장부". 어떤 후보를 노드로 인정할지(approve/reject)와 같은 개념인지(merge/alias)를 사람이 판정.

현재 규모: 논문 91 · 개념 127 · 노드 218 · builds_on 엣지 209.

## 1. 화면 — 무엇을 어떻게

상단 탭 셋: **사용법 / 지형도 / 사전**. (기본 진입 = 사용법)

### 1-A. 지형도 (메인)

#### 그래프는 어떻게 그려지나
프론트가 `/api/graph`를 부르면 백엔드가 Neo4j에서 그래프를 읽어 돌려준다(`graph_view_neo4j`). 개념은 점, `builds_on`은 선. D3 힘기반 레이아웃으로 배치. "논문 보기" 토글을 켜면 논문 노드도 함께 그린다(기본은 개념만).
- 노드 색 = 유형(ptype), 빈 원(점선) = 정의 없는 개념.

오른쪽 채팅 패널은 **[명령] / [수집]** 두 탭. + 화면 하단에 **수동 컨트롤 바**.

#### [명령] 탭 — 자연어 한 줄이 도구로 번역되는 과정
입력한 문장을 백엔드(`/api/command`)가 **필터 에이전트(gpt-5.4-mini)**에게 넘긴다. 에이전트는 그래프의 노드 이름 목록을 보고, 문장을 5가지 도구 중 하나의 "호출"로 번역해 돌려준다. **실행(하이라이트)은 프론트가** 한다.

```
사용자 문장 → 필터 에이전트(LLM) → {tool, args} → 프론트가 실행
```

도구별 메커니즘:

**① 이름 검색** — 노드 이름 일부를 치면, 프론트가 로드된 노드의 canonical을 부분 일치 검색해 강조·포커스. (LLM 안 거침, 순수 프론트.)

**② filter (조건으로 좁히기)** — `{ptype, domain, date_after}` 조건. 프론트 `applyFilter`가 **개념 노드만** 순회하며 조건에 맞는 것을 모은다(논문은 표시용이라 제외).
- 핵심: 개념의 유형/분야는 그 개념을 *정의한 논문(홈 논문)*에서 물려받는다. 그래서 "의료 분야만" = 홈 논문이 medical인 개념들.
- 시점(date_after)은 개념과 엮인 논문 중 가장 이른 arXiv 연월로 판정.

**③ focus_lineage (계보 보기)** — `lineageSets(node, 방향)`이 `builds_on` 엣지를 그래프 탐색(BFS)해 조상/자손을 모은다.
```
builds_on {from, to}:  from이 to 위에 지어짐  ⇒  to = 조상, from = 자손
조상 = to 방향으로 따라 올라감 / 자손 = from 방향으로 따라 내려감
```
주의: 노드 id(rk)는 canonical에서 하이픈→공백+소문자로 만들어진다("Self-RAG"→"self rag"). 그래서 함수는 들어온 이름을 canonical 매칭으로 노드 id에 해소한 뒤 탐색한다.

**④ semantic_search (주제로 찾기)** — 정확한 이름을 몰라도 주제를 묘사하면 의미가 가까운 노드를 찾는다.
```
질의 문장 → embed_query(임베딩) → match(코사인 유사도, 상위 8, 0.30 미만 컷)
          → 개념·논문 hit → 키 매핑 → 프론트 하이라이트
```
임베딩 비교는 백엔드에서(프론트엔 임베딩이 없으므로), 결과 노드 id만 프론트로 보내 강조. (수집 에이전트의 "현황 보고"가 쓰는 그 부품을 재사용.)

**⑤ reset** — 모든 강조·필터 해제.

> "찾아줘/보여줘"(이미 있는 것 탐색)와 "모아줘/수집"(새로 가져오기)은 다르다. 후자는 도구를 부르지 않고 "[수집] 탭에서" 안내만 한다 — 수집은 비용·파괴적이라 명령창에서 우발 실행되지 않게 막아둠.

#### [수집] 탭 — 새 논문을 지도에 더하는 과정
주제를 입력하면 **수집 에이전트(LangGraph 다단계)**가 arXiv에서 논문을 찾아 그래프에 증분 추가. 단계마다 멈춰 사람이 확인:
```
주제 입력
  → (현황 보고)   먼저 질의를 임베딩해 "이미 지도에 이런 비슷한 게 있다"를 보여줌 (세렌디피티 씨앗)
  → 해석 확인     무엇을 찾을지 확인
  → 물량 승인     arXiv 검색 결과 중 몇 편을 들일지 사람이 승인
  → gate          논문 유형 판정(서베이 등 거름)
  → extract       개념 추출
  → 보고          그래프에 쓰고(write) 결과 요약
```
세션은 `SqliteSaver`로 저장돼 새로고침에도 살아남는다(중간에 새로고침하면 [수집] 탭으로 자동 복원). 시작은 사람이 [시작]을 눌러야 한다.

#### 수동 컨트롤 — 채팅 없이 마우스로
채팅이 강력해도 "뭘 칠 수 있는지" 알아야 하니까, 열거 가능한 조작은 수동 UI로도 연다.
- **filter 컨트롤 바**(하단): 유형/분야/시점 드롭다운 + [초기화]. 옵션은 로드된 노드에서 *동적 생성*(그래프 실제 값과 항상 일치).
- **노드 클릭 계보**: 개념 노드를 클릭하면 디테일 패널에 [조상]/[자손]/[양쪽] 버튼.
- 상태 모델: `filterState`가 filter 차원의 단일 진실원. 드롭다운·채팅 filter·초기화가 모두 한 함수(`setFilter`)를 거쳐 **양방향 동기**된다("의료 분야만" 치면 드롭다운도 medical). 계보·의미검색은 자기 하이라이트를 직접 세팅하고 드롭다운 표시만 비운다(서로 안 싸움).

### 1-B. 사전 (Lexicon)
개념의 상태 장부를 보고 편집. status(approved/unreviewed/pending/rejected)·정의·alias·출처.
- **거르기**: 후보 개념을 approve/reject. reject하면 노드·엣지 삭제.
- **병합(merge)**: 같은 개념의 다른 표기를 하나로 흡수(alias).
- 원리: 개념 정의의 **정본은 논문 추출**이다. 사전은 *자격 판정*(인정 여부)과 *동일성 판정*(같은 개념인지)만 한다. 사전에서 정의를 고치는 건 임시 오버레이일 뿐, 재빌드하면 추출 정의로 돌아간다(버그 아님 — 올바른 모델).

### 1-C. 사용법 (Usage)
처음 보는 사람을 위한 페이지. "하고 싶은 것"(평범한 말)별 예시 칩. 칩을 누르면 지형도로 이동해 그 질문이 **자동 실행**되어 맵이 반응한다(읽기 전용 예시는 즉시 실행, 수집 예시는 [수집] 탭에 주제만 채워 사람 확인).

## 2. 백엔드 원리

### 2-A. 빌드 파이프라인 — 논문 하나가 그래프에 오르기까지
```
fetch     arXiv에서 PDF 가져옴
  ↓
parse     abstract + intro만 추출 (References부터 잘라냄)  ← 의도적 제약
  ↓
extract   개념 추출            (gpt-5.4-mini)
  ↓
relate    builds_on 계보 추출  (gpt-5.4 full)
  ↓
normalize_v2   표기 정규화·중복 병합 → 노드/엣지 확정
  ↓
embed     개념·논문 임베딩 (text-embedding-3-small)
  ↓
Neo4j load     그래프 DB에 적재 + verify로 검증
```

왜 이렇게:
- **abstract+intro만 쓴다** — 전문을 안 넣어도 계보의 상당 부분이 잡히는지를 측정으로 확인했다. 못 잡는 부분(형제 논문 사슬은 본문 related work에 있음)은 *구조적 한계*로 정직하게 문서화(개선 대상이 아니라 트레이드오프).
- **모델 분담**(extract=mini / relate=full) — 계보 추출(relate)은 full이 mini보다 정밀도 +0.20이라 full로 승격, 개념 추출(extract)은 mini로 충분해 비용을 아낀다. 둘 다 *측정으로* 결정.
- **builds_on = lineage-only** — 방법적 후예만 계보로 본다. 점수 비교 baseline, 부품(PPO/GRPO 등), 데이터셋, 툴은 제외. 이 규칙이 노드 정확도보다 중요한 "위상(관계 구조)"의 신뢰도를 만든다.

### 2-B. 이중 임베딩 — 두 종류의 "가까움"
- **개념** → 그 *정의*로 임베딩. "이 주제 관련 기법이 뭐 있나"에 답.
- **논문** → 그 *문제(problem)*로 임베딩. "같은 문제를 다룬 논문이 뭐 있나"에 답(세렌디피티 씨앗).
- 코사인 유사도로 비교. `semantic_search`와 수집 에이전트 현황 보고가 같은 부품을 공유.

### 2-C. 그래프 저장·재빌드·키 공간
- **단일 진입점**: 모든 Neo4j 접근은 `graphdb/conn.py`의 `get_driver()` 하나로.
- **재빌드**: 정본(추출 결과 + 사전) → `normalize_v2` → `load` → `verify`. 라이브 그래프가 증분 편집으로 틀어져도 정본에서 다시 만들어 복구.
- **키 공간이 둘**(헷갈리기 쉬운 곳):
  ```
  임베딩/매칭:   concept:<rk> · paper:<id>
  프론트 그래프: <rk>(개념, 접두사 없음) · paper:<id>(논문)
  ```
  semantic 결과를 하이라이트할 땐 개념 키에서 "concept:"를 떼어 프론트 id로 맞춘다.

## 3. 평가 — P/R를 어떻게 재나 (프로젝트의 핵심 역량)

이 프로젝트의 차별점은 "기능"보다 **"직관이 아니라 측정으로 결정했다"**는 데 있다.

```
골든셋 50편   사람이 builds_on 계보를 직접 라벨링 (정답지, freeze)
  ↓
score_buildson.py   파이프라인 출력 vs 정답지 대조 → 정밀도/재현율
  ↓
라이브 그래프(full) 게이트:  정밀도 0.82 / 재현율 0.83
```
- **정밀도(precision)** = 파이프라인이 그은 계보 중 진짜 비율. **재현율(recall)** = 진짜 계보 중 파이프라인이 잡은 비율.
- `ATLAS_OFFLINE` 모드로 평가를 라이브 그래프와 완전 격리(무오염) — eval이 JSON만 직독.
- 측정으로 내린 결정들: mini→full 승격(정밀도 +0.20), evidence·related-work 실험은 측정 후 *미채택*(개선 없거나 음성 결과 → 그 자체가 "왜 abstract-only인가"의 정당화).

### 이 수치의 한계 (정직하게)

P0.82/R0.83은 강한 신호지만 **out-of-sample 일반화 수치는 아니다.** 세 가지를 분명히 둔다.

1. **In-sample · 자기 라벨**: 정답지(`labels.json`)를 프로젝트 본인이 작성했고, 프롬프트·lexicon을
   튜닝하며 같은 50편을 반복해서 봤다. held-out(튜닝 중 한 번도 안 본) 분리가 없으므로 이 P/R는
   "내 라벨과의 일치도"이지 "처음 보는 논문에서의 성능"이 아니다.
2. **스모크는 정답이 아니라 현 동작의 스냅샷을 고정**: `score_buildson.py`의 `SMOKE`는 5편의
   TP/FP/FN 기대값을 하드코딩한다. 그래서 lexicon/프롬프트가 *개선*되면 스모크가 깨지고 수동
   재기준선이 필요하다 — 실제로 지금 KG-R1은 `KG-RAG` 별칭이 노드로 해소되며 TP=2(더 정확)로
   올랐는데, 핀은 아직 TP=1이라 스모크가 FAIL이다(개선이 회귀로 잡히는 전형적 in-sample 함정).
3. **로드맵**: 확장 라벨 85편(`survey_sourced_b2`)을 같은 규칙으로 채워 freeze한 뒤,
   **튜닝에 한 번도 쓰지 않은 held-out 세트로 out-of-sample 채점**을 하는 게 다음 단계다(이때
   frozen 50 baseline은 비교 기준으로 보존). 그 수치가 나오기 전까지 0.82/0.83은 in-sample로 읽어야 한다.
